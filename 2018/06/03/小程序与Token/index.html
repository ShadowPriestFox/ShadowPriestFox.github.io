小程序与Token

Posted on 2018-06-03

之前有提到小程序不支持cookie,我们采用的是JWT来代替cookie。
在最开始的时候是有讨论过两种方案的：

  1. 直接使用微信提供的unionId(openId)
  2. 使用token代替

两种方式各有优劣。
首先使用unionId(openId)作为区分用户的标志，好处显而易见，非常简单，流程大概如下：

  1. 用户打开小程序，调用wx.login方法获取code。
  2. 将code发送至后端，后端通过微信提供的API换取用户的unionId(openId)
  3. 将换取到的unionId(openId)传回至小程序端保存下来
  4. 每次请求后端数据时带上该Id

几乎不需要自己做额外的操作。唯一要做的是后端需要提供一个提交code换取unionId(openId)的接口。
缺点：

  1. 后端提供的接口没法校验前端的请求是否是真的用户。另外还好小程序是强制要求https的。不然拿着用户id这么来回传很容易就泄露了。(在平时面试中还遇到有说把unionId作为一个pram放到url里的。。)
  2. 无法做多平台扩展，如果以后加入App端。后续的校验，以及分多渠道做起来会比较恶心。
  3. 无法过期。一般来说cookie是会有过期时间的。。但使用id的话将没法过期。

考虑到以上这些最后还是决定使用token来代替cookie。
既然使用token的话就使用比较有名的JWT好了。库比较多，用起来也很简单，大体流程如下：

  1. 用户打开小程序，调用wx.login方法获取code。
  2. 将code发送至后端，后端通过微信提供的API获取用户的unionId(openId)
  3. 将换取到的unionId(openId)进行加密以及你想加入的相关的非敏感信息加入到相应的payload中通过JWT相应的加密算法得到Token。JWT的payload只会进行base64并不会加密，所以不要放敏感信息。
  4. 将Token放置在response Header中的Authorization: Bearer {token}中返回给小程序端
  5. 小程序端保留此token，每次请求的时候也将其放置在Authorization: Bearer 中发送给后端进行校验

这里具体实现的时候大概是这样的，首先对于需要做认证的接口加入interceptor。在preHandle 中进行校验，如果说在返回结果时同时将更新的token传至前端不能在interceptor中添加header，因为此时response已经在HandlerAdapter中被提交，如果说需要修改header需要实现ResponseBodyAdvice并声明为一个Controller Advice文档这里有提到相关信息
