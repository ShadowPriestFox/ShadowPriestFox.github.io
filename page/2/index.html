Spring Bean Scope

Posted on 2018-06-14 | Visitors:

最近在面试的时候跟候选人聊到Spring Bean的Scope，遂查了一番文档，大概总结一下。

Spring 提供6种类型的scope，其中4种是在Web下使用的，当然scope也是可以自定义的。主要有下面这些scope:

Scope 定义
singleton 默认scope，顾名思义这里的Bean是单例(在每一个Spring IoC 容器中)的。这里的单例并不是线程安全的，所以如果是Lazy Init的话必须小心。
prototype 每次使用Bean都会创建
request 每一个HTTP请求会创建一个Bean
session 每个HTTP Session会创建一个Bean
application 每个Servlet Context会创建一个Bean
websocket 每个Websocket连接会创建一个Bean

下面会对每一个scope有一个小的总结：

singleton

这是默认的scope，每一个Spring IoC容器只有一个。在需要被注入的时候会由Spring 容器注入。所以一般来说singleton的bean应该为无状态的。

prototype

这种bean在被使用的时候都是初始化一个全新的bean交由使用者使用。这里需要注意的是Spring IoC容器并不会管理prototype整个生命周期。Spring IoC在初始化的时候会调用相应的生命周期回调。一旦bean交由使用者之后Spring将不会管理该bean。也就是说如果你申明了prototype的bean在使用完之后需要手动释放相应的资源。或者需要自定义bean post-processor去做相应的处理。

singleton与prototype

公司比较老的Web项目是用Tomcat+SpringMVC来做的，有一个web.xml配置大概如下

当时看到的时候还是比较困惑的，为什么会有两个context，以及这两个context到底是干什么用的。于是上网搜索了一番。

  1. 第一个context也就是样例里的AppContextConfig，是Root WebApplicationContext。
  2. servlet标签中的context也就是样例中DispatcherConfig是servlet独享的context。

需要说明的是一般来说Services,Repositories一般是放在Root Context中的，而Controller,ViewResolver,HandlerMapping一般是放在Servlet Context中的，Root Context会被所有的Servlet Context共享，是Servlet Context的Parent。当Spring在Servlet Context中查找不到Bean时会去Root Context中查找，反过来则不会。所以一般在Controller中做AOP无法成功。

有些人认为StackOverFlow在很多情况下你其实并不需要两个Context，因为通常只会有一个Servlet，所以ContextLoaderListener和Root Context并不需要，全部使用一个就好了。这里有个地方需要注意，就是你不能把ContextLoaderListener的context与Servlet context设置成同一个，这样bean会初始化两次(一次为Root Context的，一次Servlet Context的)，如果只想使用一个的话不设置Root Context就好了。还有一点有意思的是Spring Boot就只有一个Context。

另外一种观点就较为温和一些除了以下这些情况以外使用同一个context还是比较简单的:

  1. 多个Servlet共享Service
  2. 非Spring servlet项目需要访问Spring 提供的Service
  3. 如果有Servlet filter需要(比如Spring Security的DelegatingFilterProxy,OpenEntityManagerInViewFilter等)

小程序与Token

Posted on 2018-06-03 | Visitors:

之前有提到小程序不支持cookie,我们采用的是JWT来代替cookie。
在最开始的时候是有讨论过两种方案的：

  1. 直接使用微信提供的unionId(openId)
  2. 使用token代替

两种方式各有优劣。
首先使用unionId(openId)作为区分用户的标志，好处显而易见，非常简单，流程大概如下：

  1. 用户打开小程序，调用wx.login方法获取code。
  2. 将code发送至后端，后端通过微信提供的API换取用户的unionId(openId)
  3. 将换取到的unionId(openId)传回至小程序端保存下来
  4. 每次请求后端数据时带上该Id

几乎不需要自己做额外的操作。唯一要做的是后端需要提供一个提交code换取unionId(openId)的接口。
缺点：

  1. 后端提供的接口没法校验前端的请求是否是真的用户。另外还好小程序是强制要求https的。不然拿着用户id这么来回传很容易就泄露了。(在平时面试中还遇到有说把unionId作为一个pram放到url里的。。)
  2. 无法做多平台扩展，如果以后加入App端。后续的校验，以及分多渠道做起来会比较恶心。
  3. 无法过期。一般来说cookie是会有过期时间的。。但使用id的话将没法过期。

考虑到以上这些最后还是决定使用token来代替cookie。
既然使用token的话就使用比较有名的JWT好了。库比较多，用起来也很简单，大体流程如下：

  1. 用户打开小程序，调用wx.login方法获取code。
  2. 将code发送至后端，后端通过微信提供的API获取用户的unionId(openId)
  3. 将换取到的unionId(openId)进行加密以及你想加入的相关的非敏感信息加入到相应的payload中通过JWT相应的加密算法得到Token。JWT的payload只会进行base64并不会加密，所以不要放敏感信息。
  4. 将Token放置在response Header中的Authorization: Bearer {token}中返回给小程序端
  5. 小程序端保留此token，每次请求的时候也将其放置在Authorization: Bearer 中发送给后端进行校验

这里具体实现的时候大概是这样的，首先对于需要做认证的接口加入interceptor。在preHandle 中进行校验，如果说在返回结果时同时将更新的token传至前端不能在interceptor中添加header，因为此时response已经在HandlerAdapter中被提交，如果说需要修改header需要实现ResponseBodyAdvice并声明为一个Controller Advice文档这里有提到相关信息

技术点

Java,Spring,Spring-Websocket,小程序,Websocket相关API以及JWT和STOMP

流程

  1. 用户打开小程序，调用wx.login得到code
  2. 小程序将code发给后端，后端使用code调用微信提供的接口换取用户的openid和unionid(这里只有绑定微信平台才能拿到unionid)

  4. 前端后续所有交互都需要使用该token(包括在使用Websocket和后端交互时)

JWT

小程序不允许使用cookie，所以我们将使用JWT来代替cookie。JWT没什么好说的，提供一个加密算法以及一个解密算法就好
